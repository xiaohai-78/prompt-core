# AI 编码总则

所有输出的技术文档需要使用中文。
对于不清楚、不确定的地方，必须使用搜索引擎或可用工具去证实，而不是猜测。

## 一、思维原则

每次任务开始前，明确回答三个问题：
- **为什么做**：解决什么问题，不做会怎样
- **怎么做**：核心方案和关键决策点
- **影响什么**：变更波及的所有生产者和消费者

## 二、执行流程（四阶段 + 强制检查点）

### 阶段 1：规格书（设计文档）

在动手写任何代码之前，必须完成以下工作并向用户展示：

#### 1.1 状态空间枚举（强制）

对于涉及控制流、分支、并发、异常处理的改动，必须构建**完备决策矩阵**：

```
列出：
- 所有执行路径（单路径 / 多路径 / 并发 / 嵌套）
- 每条路径的所有可能结果（成功 / 部分失败 / 全部失败 / 超时 / 异常冒出）
- 每个 (路径, 结果) 组合的期望行为（上游应看到什么、下游应收到什么）
- 标注当前代码是否已覆盖，未覆盖的 = 必须实现的
```

不构建矩阵就开始编码 = 违规。简单的单点改动（改文案、改常量）可跳过。

#### 1.2 影响面扫描（强制）

对于修改数据格式、Key 模式、接口签名、状态机的改动：

```
1. grep 所有【写入方】（谁生产这个数据）
2. grep 所有【读取方】（谁消费这个数据）
3. grep 所有【清理方】（谁删除 / 过期这个数据）
4. 逐一确认每个消费者和清理方是否兼容新格式
5. 将不兼容的列入改动清单
```

只改了生产者就认为完成 = 违规。

#### 1.3 设计文档输出

将上述分析形成设计文档（或追加到已有提案文档），包含：
- 决策矩阵
- 影响面清单
- 改动文件列表

### 阶段 2：代码编码

#### 2.1 对称性检查规则（强制）

每修改一处代码时，必须检查其对称结构：

| 修改了什么 | 必须同时检查什么 |
|---|---|
| if 分支 | else / elif 分支 |
| try 块 | except 块和 finally 块 |
| 单路径逻辑 | 并发路径逻辑 |
| 正常路径 | 异常 / 超时 / 取消路径 |
| 数据写入 | 数据读取和数据清理 |
| 新增字段 | 所有序列化 / 反序列化位置 |

修了一侧不看另一侧 = 违规。

#### 2.2 失败优先编码

编码顺序要求：
1. 先写异常 / 失败 / 边界路径
2. 再写正常路径
3. 最后写优化路径

这是为了对抗大模型"沿 happy path 写完就停"的倾向。

#### 2.3 文档同步

代码文件顶部应存放对应文档路径，文档是索引和永久记忆。
每次代码变更必须同步更新对应的设计文档。

### 阶段 3：自验（代替等用户 review）

编码完成后，必须执行以下自验，不可跳过：

#### 3.1 矩阵回验

逐行对照阶段 1 的决策矩阵：
- 每个格子在代码中是否有对应处理？指出具体代码行号
- 无法对应 = 遗漏，立即补充

#### 3.2 影响面复查

对照阶段 1 的影响面清单：
- 每个消费者 / 清理方是否已更新？
- 每个相关测试的断言是否仍然匹配当前实现？不匹配的测试 = 必须同步更新

#### 3.3 边界场景走查

用以下清单逐项检查（适用项打勾，不适用标 N/A）：

```
- [ ] 输入为空 / None
- [ ] 单元素 vs 多元素
- [ ] 全部成功 vs 部分失败 vs 全部失败
- [ ] 异常是否会冒出跳过关键清理 / 通知逻辑
- [ ] 并发竞态：同一资源是否有多个写入方
- [ ] 幂等性：重复执行是否安全
- [ ] 超时：长时间未返回的处理
```

#### 3.4 向用户报告

自验完成后，向用户报告：
- 决策矩阵的覆盖情况（全部覆盖 / 有哪些需要用户确认的设计决策）
- 影响面是否全部处理
- 边界场景走查结果

### 阶段 4：重构与修复

- 重构修复后的变化必须同步到规格书 / 设计文档中
- 如果修复引入了新的分支，回到阶段 3 重新自验

## 三、认知偏差对抗规则

以下是大模型高频犯错模式，每次编码时主动对照：

| 偏差 | 表现 | 强制对策 |
|---|---|---|
| Happy path 偏好 | 只写成功路径，遗漏失败路径 | 阶段 2.2 失败优先编码 |
| 局部满足感 | 改完一个点就认为完成 | 阶段 3.1 矩阵回验 |
| 注意力衰减 | 长文件中遗忘前面读过的信息 | 对称性检查时必须重新读取对称分支的代码 |
| 生产者思维 | 只改写入不改读取 | 阶段 1.2 影响面扫描 |
| 对称遗忘 | 改 if 不看 else | 阶段 2.1 对称性检查 |
| 测试惯性 | 实现改了但不检查测试是否过期 | 阶段 3.2 影响面复查 |

## 四、用户的四个远见

- **产品远见**：这个改动如何影响用户体验
- **架构远见**：这个改动是否引入新的耦合或破坏现有分层
- **技术远见**：这个方案在极端负载 / 异常场景下是否健壮
- **演进远见**：这个改动是否为未来扩展留有余地
